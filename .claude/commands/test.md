기능 코드를 리뷰하고 개선점을 찾아 수정해줘.

## 입력

- `$ARGUMENTS`: 리뷰할 기능명 (선택)
  - 비어있으면 전체 기능 리뷰
  - 기능명이 주어지면 해당 기능만 집중 리뷰

## 프로세스

### 기능명이 주어진 경우 (`$ARGUMENTS`가 비어있지 않은 경우)

1. `.claude/product/feature/README.md`에서 해당 기능을 찾는다.
2. 상세 기획 문서가 있으면 읽어서 요구사항을 파악한다.
3. 해당 기능과 관련된 코드 파일을 탐색한다 (services, hooks, components, app 등).
4. 아래 리뷰 체크리스트를 기준으로 코드를 분석한다.
5. 발견된 개선점을 목록으로 정리하여 사용자에게 보여준다.
6. 사용자가 승인하면 코드를 수정한다.

### 기능명이 없는 경우 (전체 리뷰)

1. `.claude/product/feature/README.md`에서 ✅ 완료된 기능 목록을 파악한다.
2. 최근 변경된 파일 위주로 리뷰 대상을 선정한다 (`git diff --name-only HEAD~5`).
3. 아래 리뷰 체크리스트를 기준으로 코드를 분석한다.
4. 발견된 개선점을 기능별로 그룹핑하여 사용자에게 보여준다.
5. 사용자가 승인하면 코드를 수정한다.

## 리뷰 체크리스트

### 성능

- [ ] 불필요한 리렌더링이 없는가 (useCallback, useMemo 적절히 사용)
- [ ] 무거운 연산이 컴포넌트 렌더 사이클 밖에 있는가
- [ ] DB 쿼리가 효율적인가 (불필요한 SELECT \*, 인덱스 누락 등)
- [ ] 리스트 렌더링에 keyExtractor가 적절한가

### 안정성

- [ ] 에러 핸들링이 적절한가 (try-catch, 사용자 피드백)
- [ ] null/undefined 체크가 필요한 곳에 있는가
- [ ] 비동기 작업의 로딩/에러 상태를 관리하는가
- [ ] 메모리 누수 가능성이 없는가 (cleanup, 언마운트 후 setState 등)

### 코드 품질

- [ ] 중복 코드가 없는가
- [ ] 타입이 적절히 정의되어 있는가 (any 남용 없는지)
- [ ] 사용되지 않는 import, 변수, 함수가 없는가
- [ ] 네이밍이 명확하고 일관적인가

### 기획 정합성

- [ ] 기획 문서의 요구사항이 모두 구현되었는가
- [ ] 예외 처리가 기획 문서와 일치하는가
- [ ] UI가 화면 구성 명세와 일치하는가

## 출력 형식

개선점을 다음 형식으로 정리:

```
## 리뷰 결과: [기능명]

### 🔴 수정 필요 (버그/오류)
- [파일:라인] 설명

### 🟡 개선 권장 (성능/품질)
- [파일:라인] 설명

### 🟢 참고 사항
- 설명
```

수정 진행 여부를 사용자에게 확인한 후 실행한다.
